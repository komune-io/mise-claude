# claude-env: Standalone CLI for Declarative Claude Code Environment Management

**Date:** 2026-04-20
**Status:** Approved
**Replaces:** mise-claude Lua backend plugin

## Problem

The current mise backend plugin approach has fundamental issues:

1. **Scope mismatch** — mise is a version manager; claude-env does config management (`.mcp.json`, `.claude/settings.json`). The abstraction doesn't fit.
2. **Fragility** — npm dependency issues (tinyexec), `pwd` assumptions, errors swallowed by pcall.
3. **Concurrency/locking** — mkdir spinners, stale locks, race conditions from mise's parallel execution model.

## Solution

A standalone Rust CLI (`claude-env`) that reads a single config file and orchestrates Claude Code tool setup. Sequential execution eliminates locking. Direct file writes eliminate the scope mismatch. Rust eliminates the fragility.

## Config Format

File: `claude-env.toml` at project root.

```toml
[mcp]
context7 = "2.1.4"
chrome-devtools = "1.0.3"
shadcn = "0.2.1"

[skills]
"vercel-labs/next-skills/next-best-practices" = "latest"
"anthropics/claude-skills/web-design-guidelines" = "latest"

[plugins]
"anthropics/claude-code/code-review@claude-code-plugins" = "latest"
"anthropics/claude-code/feature-dev@claude-code-plugins" = "latest"
"obra/superpowers/superpowers@superpowers" = "latest"

[cli]
get-shit-done-cc = "1.22.4"
bmad-method = "0.5.0"
```

Sections define tool type. No prefix parsing needed.

Short aliases (e.g., `context7` → `@upstash/context7-mcp`) resolved via built-in registry. Full npm package names also accepted directly (e.g., `"@someorg/my-custom-mcp" = "1.0.0"`).

## Lockfile

File: `claude-env.lock` at project root. Committed to git.

```toml
# Auto-generated by claude-env. Do not edit.

[mcp.context7]
package = "@upstash/context7-mcp"
version = "2.1.4"
integrity = "sha512-abc123..."

[mcp.chrome-devtools]
package = "chrome-devtools-mcp"
version = "1.0.3"
integrity = "sha512-def456..."

[skills."vercel-labs/next-skills/next-best-practices"]
version = "1.0.0"
resolved_at = "2026-04-20"

[plugins."anthropics/claude-code/code-review@claude-code-plugins"]
version = "latest"
resolved_at = "2026-04-20"

[cli.get-shit-done-cc]
package = "get-shit-done-cc"
version = "1.22.4"
integrity = "sha512-ghi789..."
```

- Pins exact resolved versions even when config says `"latest"`.
- `integrity` (sha512) for npm packages — detects supply chain tampering.
- `resolved_at` for non-npm tools (no integrity hashes available).
- `claude-env install` uses lockfile; `claude-env update` refreshes it.

## CLI Commands

```
claude-env install          # Install from lockfile (or resolve + create lockfile if missing)
claude-env update           # Check all tools for updates, show changelogs, update lockfile
claude-env update <tool>    # Update a single tool
claude-env diff <tool>      # Show changelog between pinned and latest version
claude-env list             # Show installed tools, versions, available updates
claude-env add <tool>       # Add tool to config (detects type, queries versions)
claude-env remove <tool>    # Remove from config + clean generated config files
```

### Update UX

```
$ claude-env update

  mcp/context7        2.1.4 → 2.2.0
    • Added Redis caching support
    • Fixed timeout on large repos

  cli/get-shit-done   1.22.4 → 1.23.0
    • New --parallel flag for task execution

  skills, plugins     ✓ up to date

Update all? [Y/n/select]
```

Changelogs sourced from npm registry metadata + GitHub releases API.

## Install Mechanics

Sequential pipeline on `claude-env install`:

1. **Read** `claude-env.toml` + `claude-env.lock`
2. **Resolve** — determine what needs doing (fresh install, upgrade, already satisfied)
3. **Install each tool in order:**

| Type | Action | Output |
|------|--------|--------|
| **mcp** | `npm install <pkg>@<ver> --prefix ~/.claude-env/packages/<name>` | Write entry to `.mcp.json` |
| **cli** | Same npm install | Run `post_install`, skip `.mcp.json` |
| **skills** | `npx skills add <owner/repo> --skill <name> -a claude-code -y` | Skills in `.claude/skills/` |
| **plugins** | `claude plugin marketplace add` + `claude plugin install --scope project` | Registered in `.claude/settings.json` |

4. **Write** updated `claude-env.lock`
5. **Report** results

### Key Design Decisions

- **Global package cache at `~/.claude-env/packages/`** — no per-project `node_modules`. `.mcp.json` points to absolute paths.
- **Sequential execution** — single process, one tool at a time. No locking needed.
- **Idempotent via lockfile + installed state** — compares lockfile version against `~/.claude-env/packages/<name>/package.json` (npm) or presence of expected output files (skills/plugins). Skip if satisfied. No marker files.
- **Direct `.mcp.json` writes** — read, merge, write. No concurrent writers.
- **`post_install` runs immediately** — PATH includes `~/.claude-env/packages/<tool>/node_modules/.bin`. No deferred execution.

## Tool Registry

Built into the binary. Maps aliases to npm packages and defines per-tool overrides.

**Aliases:**
| Alias | Package |
|-------|---------|
| context7 | @upstash/context7-mcp |
| chrome-devtools | chrome-devtools-mcp |
| shadcn | shadcn |
| gsd | get-shit-done-cc |
| bmad | bmad-method |
| openspec | @fission-ai/openspec |

**Overrides (post_install, bin_name, extra_deps):**

| Package | Override |
|---------|----------|
| shadcn | `bin_name = "shadcn"`, `post_install = "shadcn mcp init --client claude"`, `extra_deps = ["tinyexec@1.0.2"]` |
| get-shit-done-cc | `post_install = "get-shit-done-cc --claude --local"` |
| bmad-method | `post_install = "bmad-method install --directory ${PROJECT_ROOT} --modules bmm --tools claude-code --yes"` |
| @fission-ai/openspec | `post_install = "openspec init --tools claude"` |

Unknown tools (not in registry) work directly — use the full npm package name in config.

## Error Handling

- **Fail per-tool, not globally** — one broken tool doesn't stop others.
- **Show actual errors** — no swallowing, full stderr from failed commands.
- **Exit codes** — 0 = success, 1 = partial failure, 2 = config error.
- **`--verbose`** — shows exact commands being run.
- **Known-issue hints** — registry can surface known npm breakage.

```
$ claude-env install

  ✓ mcp/context7         2.1.4 installed
  ✓ mcp/chrome-devtools  1.0.3 installed
  ✗ mcp/shadcn           0.2.1 failed
    │ npm install exited with code 1
    │ Error: tinyexec@1.0.3 has no valid main export
    │ Hint: known issue, try `claude-env update shadcn`
  ✓ cli/gsd              1.22.4 installed + post_install ran
  ⊘ skills/next-best-practices  skipped (already installed)

  3 installed, 1 failed, 1 skipped
```

## Testing Strategy

Three layers: unit, integration, end-to-end.

### Unit Tests

Pure Rust tests (`#[cfg(test)]`) covering internal logic without I/O:

- **Config parsing** — valid TOML → struct, invalid TOML → clear error, missing sections tolerated, unknown keys rejected
- **Alias resolution** — known alias → package name, unknown alias → pass-through, scoped packages (`@org/pkg`) → unchanged
- **Lockfile parsing/serialization** — round-trip fidelity, handles missing fields gracefully
- **Version comparison** — semver ordering, `"latest"` resolution, pre-release filtering
- **Integrity verification** — sha512 match → pass, mismatch → fail with clear message
- **Registry lookup** — override merging, `extra_deps` expansion, `${PROJECT_ROOT}` substitution

### Integration Tests

Test each tool type's install pipeline against real (or mocked) external services. Use `assert_cmd` + `assert_fs` crates for CLI invocation and filesystem assertions.

**With mocked HTTP (wiremock):**
- **npm resolution** — mock registry response, verify correct version selected and lockfile written
- **Changelog fetching** — mock GitHub releases API, verify `claude-env diff` output

**With real filesystem (temp dirs):**
- **MCP install** — run `claude-env install` with a config containing one MCP tool, assert:
  - Package exists in `~/.claude-env/packages/<name>/`
  - `.mcp.json` contains correct entry (command, args, type)
  - Lockfile updated with integrity hash
- **CLI install + post_install** — assert:
  - Package installed, binary callable
  - `post_install` command executed (verify side effects, e.g., directory created)
  - `.mcp.json` NOT modified
- **Skills install** — assert:
  - `npx skills add` invoked with correct args (capture via mock script)
  - Lockfile records `resolved_at`
- **Plugin install** — assert:
  - `claude plugin marketplace add` + `claude plugin install` invoked in order
  - `.claude/settings.json` contains plugin entry
- **Idempotency** — run `claude-env install` twice, second run reports all tools as "skipped"
- **Partial failure** — one tool fails, others still install, exit code = 1

**Command shims for external tools:**
Tests that invoke `npm`, `npx`, `claude` use shim scripts in a test-only PATH that record invocations to a log file. This avoids hitting real registries/CLIs while verifying correct argument construction.

### End-to-End Tests

Full pipeline against real services. Run in CI (not locally by default) with network access.

- **Matrix:** macOS arm64, Linux x86_64
- **Scenarios:**
  - Fresh install of 1 MCP + 1 skill + 1 plugin → verify all output files
  - `claude-env update` detects real version bump on a known package
  - `claude-env remove` cleans up `.mcp.json` entry and package directory
  - `claude-env add` with interactive input (simulated via stdin pipe)

### Test Infrastructure

```
tests/
├── unit/              # Pure logic tests (no I/O)
├── integration/       # CLI invocation + filesystem + mocked HTTP
│   ├── fixtures/      # Sample claude-env.toml files
│   ├── shims/         # Mock scripts for npm, npx, claude
│   └── mod.rs
└── e2e/               # Real network, real tools (Docker-only)
    ├── Dockerfile     # Node 20 + claude CLI + claude-env binary
    ├── docker-compose.yml
    └── scenarios/     # Full install/update/remove test scripts
```

**Execution model:**

| Layer | Runs where | Command | Speed |
|-------|-----------|---------|-------|
| Unit + Integration | Local + CI | `cargo test` | ~2s |
| E2E | Docker only | `docker compose run e2e` | ~30-60s |

**Why Docker for E2E only:**
- Integration tests use shims (no real tools needed) — Docker would add startup overhead for no benefit.
- E2E tests require real `npm`, `npx`, `claude` CLI at specific versions — Docker guarantees consistent environment across machines.
- Network isolation at container level enables testing offline/degraded scenarios.

**CI setup:**
- Every push/PR: `cargo test` (unit + integration, fast, no network)
- Merge to main + release: `docker compose run e2e` (full pipeline, real tools)
- Coverage target: 80%+ on unit/integration combined

## Non-Goals

- Not a version manager — doesn't manage Node, Rust, etc.
- Not a task runner — no `claude-env run`.
- No concurrent/parallel installs — simplicity over speed.
- No plugin system — the registry is built-in. External tools use full package names.
